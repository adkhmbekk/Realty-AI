# ADR-0001: Гибридная асинхронность (async-маршруты + httpx, БД через threadpool)

Статус: принято (аудит, Этап 13). Дата: 2026-06.

## Контекст
Долгие сетевые операции (импорт фото из Telegram до 20 шт., отправка медиагруппы/шаринг) выполнялись синхронно внутри HTTP-обработчиков (находка H3), что подвешивало воркеры и UI (M12), особенно на латентности ngrok.

## Решение
Гибридный подход вместо полного перевода слоя данных на asyncpg:
- HTTP-обработчики с тяжёлым сетевым I/O делаем `async def`.
- Внешние HTTP-вызовы (Telegram Bot API, выкачивание фото) — через `httpx.AsyncClient` с таймаутами и ограниченной конкуренцией.
- Доступ к БД остаётся синхронным (SQLAlchemy Session) и вызывается из async-маршрутов через `run_in_threadpool`/`anyio.to_thread`, чтобы не блокировать событийный цикл.

## Почему не полный asyncpg + AsyncSession
Полный перевод переписывает все репозитории/сервисы и заново затрагивает проверенные сильные стороны (изоляция арендаторов, атомарный `display_id`, транзакции) — высокий риск прямо перед первым клиентом. Гибрид решает корневую проблему H3 (блокирующий сетевой I/O) при минимальном радиусе изменений.

## Последствия
- Плюс: неблокирующий сетевой I/O, таймауты и отменяемость, сохранён проверенный слой данных.
- Минус: смешение sync/async требует дисциплины — любой sync-вызов в async-маршруте оборачивается в threadpool; CPU-bound (Pillow) тоже в threadpool.
- Полный async на asyncpg остаётся опцией на потом (roadmap), если понадобится горизонтальное масштабирование.
