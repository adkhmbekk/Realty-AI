## 1. Цель и контекст проекта

### 1.1. Что это
Программный продукт для риелторских агентств: ведение базы объектов недвижимости (квартиры и др.), поиск по фильтрам, управление сотрудниками. Интерфейс — **Telegram Mini App** (веб-приложение, открываемое внутри Telegram).

### 1.2. Бизнес-модель
Продукт сдаётся в аренду по подписке (SaaS). Каждое агентство платит ежемесячную плату. При неоплате доступ замораживается, данные сохраняются.

### 1.3. Отправная точка
Существует рабочий Telegram-бот (`realty_bot`), однопользовательский (под одно агентство), на Python + PostgreSQL, в режиме polling. Его функциональное ядро (модель данных, генерация ID, CRUD, поиск, архив, аудит) переиспользуется. Диалоговый интерфейс (FSM, клавиатуры) заменяется веб-интерфейсом.

### 1.4. Ключевые отличия от текущего бота
- Из однопользовательского → в **мультитенантный** (много агентств на одной системе).
- Из FSM-диалогов → в **веб-интерфейс (Mini App) + REST API**.
- Захардкоженные агенты и районы → **настройки каждого агентства**.
- Появляются **подписки, биллинг, роли, приглашения сотрудников**.
- Фото объектов хранятся в системе, а не как ссылка на внешний диск.

## 3. Принципы проектирования (модульность)

Главное требование заказчика: проект должен быть **гибким и модульным** — каждая часть отвечает только за себя, вместе работают слаженно. Закрепляем это как обязательные правила:

1. **Разделение слоёв (layered architecture).** Жёстко разделяем:
   - **API/Presentation** — приём HTTP-запросов, валидация формата, отдача ответов. Не содержит бизнес-логики.
   - **Services (бизнес-логика)** — правила домена (генерация ID, перевод в архив, проверка подписки). Не знает про HTTP и про SQL напрямую.
   - **Repositories (доступ к данным)** — единственный слой, который пишет SQL/работает с ORM. Бизнес-логика обращается к данным только через него.
   - **Models/Schemas** — описание сущностей и контрактов вход/выход.
2. **Каждый домен — отдельный модуль.** `apartments`, `agencies`, `users`, `subscriptions`, `dictionaries`, `auth`, `audit` — отдельные пакеты со своими роутами, сервисами, репозиториями, схемами.
3. **Изоляция арендаторов — на уровне фундамента.** Доступ к данным агентства проходит через единый механизм, который автоматически подставляет `agency_id`. Невозможность «забыть фильтр» закладывается архитектурно (см. раздел 6).
4. **Зависимости направлены внутрь.** Внешние детали (Telegram, конкретная БД, хранилище фото) скрыты за интерфейсами, чтобы их можно было заменить без переписывания бизнес-логики.
5. **Конфигурация — только из окружения** (как в текущем `config.py`): без секретов в коде, fail-fast при отсутствии обязательных переменных.
6. **Безопасность по умолчанию (fail-closed):** нет доступа, пока он явно не разрешён — наследуем подход текущего бота.
7. **Тестируемость:** сервисы и репозитории покрываются юнит-тестами без подключения к Telegram.

## 4. Общая архитектура системы

### 4.1. Компоненты

```
                         ┌──────────────────────────┐
                         │        Telegram           │
                         │  (клиент пользователя)    │
                         └──────────┬────────────────┘
                                    │  открывает Mini App по кнопке бота
                                    ▼
┌──────────────┐        ┌──────────────────────┐        ┌─────────────────┐
│  Telegram-бот │◀──────▶│   Frontend (Mini App) │◀──────▶│   API (Backend)  │
│  (точка входа,│  HTTPS │   React + TG WebApp   │  HTTPS │   FastAPI        │
│  уведомления) │        │   SDK (статика)       │  REST  │                 │
└──────┬───────┘        └──────────────────────┘        └───────┬─────────┘
       │                                                          │
       │                                                          ▼
       │                                          ┌──────────────────────────┐
       │                                          │   PostgreSQL (данные)     │
       │                                          └──────────────────────────┘
       │                                                          │
       │                                          ┌──────────────────────────┐
       └─────────────────────────────────────────▶│ Объектное хранилище (фото)│
                                                   └──────────────────────────┘

      Всё за Reverse-proxy (Caddy/Nginx) с HTTPS (обязательно для Mini App).
```

### 4.2. Описание компонентов

- **Telegram-бот** — обслуживает все агентства одновременно (один общий бот). Роли: показ кнопки запуска Mini App, проверка/регистрация пользователя через приглашения, отправка уведомлений. Бизнес-логику не дублирует — ходит в тот же API/сервисы.
- **Frontend (Mini App)** — SPA на React, отдаётся как статика. Авторизуется через `initData`, общается с API по REST. Разные интерфейсы для ролей.
- **API (Backend)** — FastAPI. Реализует всю бизнес-логику и доступ к данным. Единая точка истины.
- **PostgreSQL** — хранение всех данных (мультитенантно, через `agency_id`).
- **Объектное хранилище** — фотографии объектов (S3-совместимое; для self-host — MinIO).
- **Reverse-proxy** — HTTPS-терминация, выдача TLS-сертификатов, маршрутизация.
- **(Опционально, фаза 2+)** Redis — кэш, сессии, rate-limit; очередь задач для рассылок.

### 4.3. Топология серверов
- **Фаза 1 (MVP):** один сервер (VPS), все компоненты в Docker Compose. Один бот на всех. Управляемая БД опциональна.
- **Фаза 2:** вынос PostgreSQL в managed-сервис, фото — в облачное S3, бэкапы.
- **Фаза 3:** горизонтальное масштабирование API за балансировщиком, Redis, очередь.

## 5. Роли и матрица доступа (RBAC)

### 5.1. Роли
- **`superadmin`** — владелец платформы. `agency_id = NULL`. Управляет агентствами и подписками. Не вмешивается в данные объектов агентств (кроме сервисных задач/поддержки — по отдельному явному праву, логируется).
- **`agency_admin`** — администратор агентства. Управляет сотрудниками, справочниками (агенты, районы), настройками агентства; полный доступ к объектам своего агентства.
- **`agent`** — сотрудник агентства. Работает с объектами своего агентства.

### 5.2. Матрица прав

| Действие | superadmin | agency_admin | agent |
|---|:---:|:---:|:---:|
| Создать/заморозить агентство | ✅ | ❌ | ❌ |
| Управлять подписками (продление) | ✅ | ❌ | ❌ |
| Видеть список всех агентств | ✅ | ❌ | ❌ |
| Приглашать сотрудников своего агентства | ❌ | ✅ | ❌ |
| Удалять/деактивировать сотрудников | ❌ | ✅ | ❌ |
| Редактировать справочники (агенты, районы, типы) | ❌ | ✅ | ❌ |
| Создавать объект | ❌ | ✅ | ✅ |
| Редактировать объект | ❌ | ✅ | ✅ (*) |
| Перевести в архив / восстановить | ❌ | ✅ | ✅ (*) |
| Удалить объект | ❌ | ✅ | ❌ (по умолчанию) |
| Поиск по объектам | ❌ | ✅ | ✅ |
| Просмотр журнала действий агентства | ❌ | ✅ | ❌ |
| Загрузка фото объекта | ❌ | ✅ | ✅ |

(*) — политика «агент редактирует только свои объекты или любые объекты агентства» выносится в **настройку агентства** (см. раздел 9.9). По умолчанию — агент работает со всеми объектами своего агентства.

> Все права проверяются на стороне **API** (нельзя доверять фронтенду). Фронтенд лишь прячет недоступные кнопки для удобства.

## 6. Мультитенантность и изоляция данных

### 6.1. Подход
Shared Database + `agency_id` в каждой «рабочей» таблице. Одна БД на всех, логическая изоляция.

### 6.2. Жёсткие правила
1. **Каждая рабочая таблица** (`apartments`, `apartment_photos`, `agents`, `dictionaries`, `audit_log`, и т.д.) содержит `agency_id NOT NULL`.
2. **Любой запрос на чтение/запись данных агентства обязан фильтроваться по `agency_id`.**
3. Изоляция реализуется **централизованно**: репозитории получают «контекст текущего агентства» из слоя зависимостей и **сами** добавляют условие по `agency_id`. Прикладной код не формирует `agency_id` вручную.
4. Дополнительный рубеж защиты (рекомендуется на фазе 2): **PostgreSQL Row-Level Security (RLS)** — политики на уровне БД, отсекающие чужие строки даже при ошибке в коде.
5. Идентификаторы, видимые пользователю (например `SAR-0001`), **уникальны в пределах агентства**, а не глобально (см. раздел 8 — суррогатный первичный ключ + `display_id`).

### 6.3. Запреты
- Запрещены запросы к рабочим таблицам без условия по `agency_id` (кроме явных сервисных задач суперадмина, которые логируются).
- Запрещено передавать `agency_id` с фронтенда как параметр доступа — он определяется **только** из сессии текущего пользователя на сервере.

## 7. Аутентификация и авторизация

### 7.1. Принцип
Логины/пароли **не используются**. Личность пользователя подтверждает Telegram через `initData`.

### 7.2. Поток входа (happy path)
1. Пользователь открывает Mini App из бота. Telegram передаёт фронтенду `initData`.
2. Фронтенд отправляет `initData` на `POST /api/v1/auth/telegram`.
3. Бэкенд **проверяет подпись** `initData` (HMAC-SHA256, ключ выводится из токена бота). Проверяет `auth_date` (срок свежести, напр. ≤ 24 ч) для защиты от повторного использования.
4. Бэкенд ищет пользователя по `telegram_id`:
   - найден и активен → определяет его `agency_id` и роль;
   - не найден → пользователь не привязан ни к какому агентству → доступ закрыт (см. 7.4 про приглашения).
5. Бэкенд проверяет **статус подписки агентства**:
   - активна → выдаёт сессионный JWT (короткоживущий) + refresh-механизм;
   - заморожена/истекла → возвращает специальный статус «подписка неактивна».
6. Фронтенд сохраняет токен и далее передаёт его в каждом запросе (`Authorization: Bearer ...`).

### 7.3. Содержимое сессии (JWT payload)
- `user_id`, `telegram_id`, `agency_id` (или null для суперадмина), `role`, срок действия.
- Сервер на каждом запросе валидирует токен и **повторно проверяет**, что пользователь активен и подписка агентства в силе (или кэширует это ненадолго).

### 7.4. Приглашения и привязка
- Незнакомый пользователь не получает доступ автоматически.
- Привязка происходит через **приглашение** (см. модуль 9.3): пользователь открывает приглашение (deep-link бота с кодом или ввод кода в Mini App), система привязывает его `telegram_id` к агентству с указанной ролью.

### 7.5. Безопасность
- `initData` проверяется **на каждом** первичном входе; нельзя доверять данным без проверки подписи.
- Токен бота хранится только на сервере.
- Все права — на сервере (fail-closed).
- Защита от replay: проверка `auth_date`, опционально nonce.

## 8. Модель данных (детально)

> Соглашения: первичные ключи — `BIGSERIAL` или `UUID` (суррогатные). Временные метки в UTC. Все денежные значения — с указанием валюты. Внешние ключи — с `ON DELETE` по правилам ниже. Индексы по `agency_id` на всех рабочих таблицах обязательны.

| created_at | TIMESTAMPTZ DEFAULT now() | |
| updated_at | TIMESTAMPTZ DEFAULT now() | |

### 8.2. `users` — пользователи системы (заменяет `allowed_users`)
| Поле | Тип | Описание |
|---|---|---|
| id | BIGSERIAL PK | |
| telegram_id | BIGINT UNIQUE NOT NULL | Идентификатор Telegram |

| used_at | TIMESTAMPTZ NULL | Когда использовано |
| used_by_telegram_id | BIGINT NULL | Кем использовано |

### 8.4. `subscriptions` — история подписок/платежей
| Поле | Тип | Описание |
|---|---|---|
| id | BIGSERIAL PK | |
| agency_id | BIGINT FK → agencies NOT NULL | |

| period_end | TIMESTAMPTZ NOT NULL | Конец оплаченного периода |
| amount | NUMERIC(12,2) NULL | Сумма |

### 8.5. `agents` — справочник агентов агентства (заменяет `AGENT_CODES` + `agent_counters`)
| Поле | Тип | Описание |
|---|---|---|
| id | BIGSERIAL PK | |
| agency_id | BIGINT FK → agencies NOT NULL | |
| name | TEXT NOT NULL | Имя агента (для отображения) |
| code | VARCHAR(5) NOT NULL | Короткий код для генерации ID объектов |

Ограничения: `UNIQUE(agency_id, code)`. Генерация ID объекта: `UPDATE agents SET last_number = last_number + 1 WHERE id = ... RETURNING last_number` (атомарно, как в текущем `generate_id`).

### 8.6. `dictionaries` — гибкие справочники агентства (районы, типы, состояния и т.д.)
| Поле | Тип | Описание |
|---|---|---|
| id | BIGSERIAL PK | |
| agency_id | BIGINT FK → agencies NOT NULL | |
| category | TEXT NOT NULL | `district` / `property_type` / `condition` / `furniture` / `source` ... |
| value | TEXT NOT NULL | Значение (напр. «Чиланзарский») |
| sort_order | INTEGER DEFAULT 0 | Порядок отображения |
| is_active | BOOLEAN DEFAULT true | |

Ограничения: `UNIQUE(agency_id, category, value)`.
Назначение: вынести захардкоженные `DISTRICTS` и список источников/состояний в настраиваемые справочники, индивидуальные для каждого агентства.

## 9. Функциональные требования по модулям

> Для каждого модуля: назначение, кто пользуется, ключевые сценарии, правила.

### 9.1. Модуль `auth` — аутентификация
- Проверка `initData`, выдача и обновление JWT, определение роли/агентства.
- Обработка случая «пользователь не привязан» и «подписка неактивна».
- Правила безопасности из раздела 7.

### 9.2. Модуль `agencies` — управление агентствами (суперадмин)
- Создать агентство (название, назначить первого `agency_admin` по telegram_id/username, задать срок подписки).
- Список агентств с фильтрами (по статусу, по сроку подписки).
- Заморозить / разморозить / продлить.
- Карточка агентства: данные, сотрудники (кол-во), активность, история подписок.

### 9.3. Модуль `users` + `invites` — сотрудники и приглашения
- **Админ агентства**: создать приглашение (роль `agent` по умолчанию), получить ссылку/код; список сотрудников; деактивировать/активировать; сменить роль (в пределах агентства, не выше своей).
- **Приглашённый**: открывает приглашение → привязка `telegram_id` к агентству. Проверка: код не истёк, не использован, агентство активно.
- Деактивация сотрудника не удаляет его данные/историю.

### 9.4. Модуль `subscriptions` — подписки/биллинг
- Суперадмин фиксирует оплату (период, сумма, способ) → обновляется `subscription_expires_at` и статус агентства.
- Автоматическая проверка истечения (фоновая задача/при входе): по истечении + grace-период → статус `frozen`.
- На фазе 1 — ручное проведение платежей. Автобиллинг (Payme/Click/Telegram Payments) — фаза 2+ (раздел 19).

### 9.5. Модуль `apartments` — объекты (CRUD)
- Создать объект: валидация полей, генерация `display_id` через `agents.code` + счётчик, привязка `agency_id` и `created_by`.
- Просмотр карточки.
- Редактирование: только разрешённые поля (белый список, как в текущем `_UPDATABLE_FIELDS`).
- Перевод в архив (`status='archived'`, `archived_at=now()`), восстановление, пометка «продан».
- Удаление — по правам (по умолчанию только `agency_admin`).
- Все операции изолированы по `agency_id`.

### 9.6. Модуль `search` — поиск/фильтры
- Фильтры: район(ы), тип(ы), комнаты, диапазон этажей, диапазон цены, статус (по умолчанию `active`), агент.
- Пагинация и сортировка (по дате/цене).
- Только в пределах своего агентства.
- Логика наследует `search_apartments`, но с обязательным `agency_id` и `status`.

### 9.7. Модуль `photos` — фотографии
- Загрузка фото объекта (несколько), хранение в объектном хранилище, ссылка в `apartment_photos`.
- Ограничения: размер, формат, максимум фото на объект (настраивается).
- Удаление/переупорядочивание. Удаление объекта → каскадное удаление фото (и из хранилища — фоновая очистка).

### 9.8. Модуль `dictionaries` — справочники агентства
- Управление списками: районы, типы, состояния, источники и т.д. (только `agency_admin`).
- Используются в формах объектов и фильтрах поиска.

### 9.9. Модуль `agency_settings` — настройки агентства
- Валюта по умолчанию, часовой пояс.
- Политика прав агента (может ли агент редактировать чужие объекты, может ли удалять).
- Лимиты (макс. фото на объект и т.п.).
- Хранится в `agencies.settings` (JSONB) для гибкости.

### 9.10. Модуль `audit` — журнал действий
- Запись значимых действий (create/update/archive/delete/login/invite/...).
- Просмотр журнала агентства (`agency_admin`).
- Санитизация пользовательского ввода (наследуем `safe_username`).

### 9.11. Модуль `notifications` (опционально, фаза 2)
- Уведомления через бота: «подписка истекает через N дней», «добавлен новый объект» и т.п.

## 13. Нефункциональные требования

### 13.1. Безопасность
- Параметризованные запросы везде (наследуем подход текущего проекта).
- Белые списки полей для обновления (наследуем `_UPDATABLE_FIELDS`).
- Проверка `initData`, секреты только в окружении, fail-fast конфиг.
- Все права — на сервере; фронт не источник истины.

- Rate-limit на чувствительные эндпоинты, **с учётом агентства**.
- Санитизация логов (наследуем `safe_username`).
- HTTPS обязателен.

### 13.2. Надёжность и данные
- **Автоматические бэкапы БД** (ежедневно) + проверка восстановления — обязательно с первого платящего клиента.
- Транзакционность операций «архив/восстановление».
- Грейсфул-обработка ошибок, понятные сообщения.

### 13.3. Производительность
- Индексы под фильтры поиска (раздел 8.7).
- Пагинация списков.
- Пул соединений к БД (наследуем подход `db/connection.py`).

### 13.4. Наблюдаемость
- Структурные логи, уровни (наследуем `logging_setup`).
- Мониторинг ошибок (Sentry или аналог) — фаза 2.
- Метрики здоровья сервиса (`/health`).

### 13.5. Конфиденциальность
- В системе хранятся персональные данные (телефоны). Предусмотреть удаление по запросу и недоступность между агентствами. Юридические аспекты — раздел 19.

### 13.6. Совместимость
- Mini App: актуальные версии Telegram (iOS/Android/Desktop/Web).

## 14. Структура проекта (раскладка по модулям)

> Принцип: монорепозиторий с независимыми частями. Каждая часть отвечает только за себя.

```
realty_saas/
├── backend/                       # FastAPI-приложение (API + бизнес-логика)
│   ├── app/
│   │   ├── main.py                # сборка приложения, подключение роутеров
│   │   ├── config.py              # настройки из окружения (fail-fast)
│   │   ├── core/
│   │   │   ├── security.py        # проверка initData, JWT
│   │   │   ├── dependencies.py    # current_user, current_agency
│   │   │   ├── tenancy.py         # механизм изоляции по agency_id
│   │   │   ├── permissions.py     # проверки ролей (RBAC)
│   │   │   └── exceptions.py      # единый формат ошибок
│   │   ├── db/
│   │   │   ├── session.py         # пул соединений/сессия
│   │   │   ├── models/            # ORM-модели (по сущности на файл)
│   │   │   └── migrations/        # Alembic
│   │   ├── repositories/          # доступ к данным (по сущности)
│   │   ├── services/              # бизнес-логика (по домену)
│   │   ├── schemas/               # Pydantic-схемы (вход/выход)
│   │   ├── api/
│   │   │   ├── router.py          # агрегатор роутеров
│   │   │   └── routes/            # эндпоинты по модулям
│   │   └── utils/                 # вспомогательное (sanitize, ids, time)
│   ├── tests/                     # юнит/интеграционные тесты
│   ├── requirements.txt
│   └── Dockerfile
│
├── bot/                           # Telegram-бот (точка входа, уведомления)
│   ├── main.py
│   ├── config.py
│   ├── handlers/                  # /start, deep-link приглашений, кнопка Mini App
│   ├── services/                  # обращение к API/общим сервисам
│   └── Dockerfile
│
├── frontend/                      # Mini App (React + TS)
│   ├── src/
│   │   ├── api/                   # клиент к backend
│   │   ├── telegram/              # обёртка над Telegram WebApp SDK
│   │   ├── features/              # по фичам: apartments, search, employees, agencies, subscriptions
│   │   ├── components/            # переиспользуемые UI-компоненты
│   │   ├── pages/                 # экраны по ролям
│   │   ├── hooks/
│   │   ├── store/                 # состояние
│   │   └── types/
│   ├── package.json
│   └── Dockerfile
│
├── deploy/
│   ├── docker-compose.yml         # bot + backend + frontend + db + caddy
│   ├── Caddyfile                  # reverse-proxy + авто-HTTPS
│   └── .env.example
│
└── docs/
    └── TECHNICAL_SPECIFICATION.md # этот документ
```

Соответствие принципу «каждая часть за себя»:
- **routes** не знают про SQL; **services** не знают про HTTP; **repositories** — единственные, кто пишет SQL.
- Telegram, БД, хранилище фото скрыты за слоями и заменяемы.
- Frontend организован по фичам, а не «свалкой» компонентов.

## 16. Миграции, окружения, деплой

### 16.1. Миграции
- Все изменения схемы — через **Alembic** (никаких ручных правок прод-БД).
- Стартовая миграция создаёт все таблицы из раздела 8.

### 16.2. Окружения
- `local` — Docker Compose на машине разработчика.
- `production` — один VPS (фаза 1), затем managed-сервисы (фаза 2+).
- Все секреты — в `.env` (не в репозитории), есть `.env.example`.

### 16.3. Деплой (фаза 1)
- `docker-compose up` поднимает: backend, bot, frontend (статика за Caddy), PostgreSQL, Caddy (HTTPS).
- Домен + TLS обязательны (для Mini App).
- Настройка Mini App URL и кнопки в @BotFather.

### 16.4. Резервное копирование
- Ежедневный дамп БД, хранение в отдельном месте, регулярная проверка восстановления.

## 18. Критерии приёмки

Проект считается готовым к продаже первому клиенту (MVP), когда:
1. Суперадмин может создать агентство, назначить админа, задать срок подписки.
2. Админ агентства входит через Telegram без пароля и видит только своё агентство.
3. Админ может пригласить сотрудника по ссылке; сотрудник получает доступ только к своему агентству.
4. Агент/админ могут создавать, искать, редактировать, архивировать объекты; ID генерируется корректно и уникален в пределах агентства.
5. Данные одного агентства **недоступны** другому (проверено тестом изоляции).
6. К объекту можно прикрепить фото.
7. При истечении подписки агентство замораживается, данные сохраняются, показывается понятное сообщение.
8. Все права проверяются на сервере; обход через фронтенд невозможен.
9. Настроены ежедневные бэкапы БД и проверено восстановление.
10. Развёрнуто за HTTPS, Mini App открывается из бота.

## 19. Открытые вопросы и решения на будущее

1. **Платёжный провайдер.** Для рынка (предположительно Узбекистан) типовые западные провайдеры недоступны; рассмотреть Payme/Click/Uzum и/или Telegram Payments через локального провайдера. Требует проверки актуальной доступности перед автобиллингом.
2. **Один пользователь в нескольких агентствах.** Сейчас `telegram_id` уникален (одно агентство). Если понадобится работать в нескольких — потребуется связь many-to-many (`memberships`). Решение отложено.
3. **Единая таблица объектов vs отдельный архив.** Выбрана единая таблица со `status` (проще, без дублирования). Трейд-офф: при огромном архиве может понадобиться партиционирование/вынос — решается позже.
4. **Удаление объектов.** По умолчанию запрещено агентам и реализовано как «жёсткое» удаление только для админа. Возможна «мягкая» корзина — открытый вопрос.
5. **Мультиязычность интерфейса.** Сейчас русский. При выходе на другие рынки — i18n. Заложить структуру, реализовать позже.
6. **Юридические аспекты ПДн.** Хранение телефонов/контактов: оферта, согласие, политика удаления. Требует проработки до массовых продаж.
7. **Лимиты тарифов.** Возможны тарифы с лимитами (число сотрудников/объектов/фото). Заложено через `agencies.settings`, конкретные тарифы — позже.
8. **Поддержка/импорт данных.** Перенос существующей базы текущего агентства в новую систему — отдельная задача миграции данных.
9. **RLS в PostgreSQL.** Дополнительный рубеж изоляции — внедрить на фазе 2.
