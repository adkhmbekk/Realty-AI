# Спецификация: переход на юзер-центричную модель

- **Дата:** 2026-07-10
- **Статус:** согласован дизайн, готовится план реализации
- **Автор обсуждения:** владелец продукта + Claude (брейнсторминг)
- **Прототип (кликабельный, в дизайне приложения):** `docs/superpowers/specs/2026-07-10-user-centric-pivot-prototype.html`

---

## 1. Цель

Перевернуть приложение с модели «всё вокруг агентства» на модель «всё вокруг **юзера**».
Сейчас: один юзер = одно агентство (`user.agency_id`), при первом входе приложение сразу
требует создать/открыть агентство. Нужно: центр — **личность юзера** (имя, фамилия, номер),
а агентства — то, во что юзер входит; юзер может состоять в **нескольких** агентствах с
разными ролями и переключаться между ними.

Дополнительная цель — **заложить фундамент** под будущее (сайт, мобильное приложение, вход по
SMS/паролю, элементы мессенджера/соцсети), не строя эти функции сейчас.

## 2. Границы

**В объёме этой работы:**
- Личный аккаунт юзера (профиль: имя, фамилия, номер, язык).
- Онбординг: язык → профиль + «поделиться номером» (номер можно менять).
- Личное пространство («хаб»): список агентств с ролью; «создать»/«вступить» спрятаны в «＋» и
  «Настройки», когда есть хотя бы одно агентство.
- Мультичленство: юзер в нескольких агентствах; вход/переключение/возврат.
- Обобщение существующего механизма `acting` (вход «внутрь» агентства) со суперадмина на всех.
- Модель данных и токены, готовые к сайту/приложению без переделки.
- Миграция существующих юзеров (у каждого сейчас ровно одно агентство).
- Тесты на изоляцию, роли, онбординг, миграцию.

**НЕ в объёме (на будущее, но проектируем совместимо):**
- Вход по SMS-коду (OTP) на сайте/в приложении.
- Логин+пароль как доп. способ входа.
- Подтверждение номера при смене; точечные сессии по устройствам.
- Функции мессенджера/соцсети (чат, лента, профили-визитки).

## 3. Согласованные решения (из брейнсторминга)

1. **Масштаб:** только переворот на юзера; мессенджер/соцсеть — направление на будущее.
2. **Данные CRM (клиенты/объекты) остаются внутри агентства.** Изоляция арендаторов сохраняется
   как сейчас (священный инвариант).
3. **Номер телефона:** нужен и подтверждён у **главного админа** (создателя агентства); у
   сотрудника — по желанию (чтобы админ видел контакт). Проверка номера — на будущее.
4. **Auth-модель:** внутренний аккаунт с несколькими способами входа, **номер — якорь**. Сейчас —
   только Telegram-вход + сбор номера; SMS/пароль — потом, без переделки схемы.
5. **«Синхрон» трёх платформ** не строим отдельно: одна база + один бэкенд = три окна в один
   аккаунт. Различается только шаг входа; после входа — общий JWT (уже платформо-независим).
6. **Навигация/переключение** (утв. в прототипе): «‹» — выход в личное пространство; тап по
   названию агентства — быстрый переключатель между агентствами.
7. **UI личного хаба:** нижние вкладки `Главная / Настройки / Профиль` (Профиль — справа). Дизайн —
   строго как в текущем приложении (индиго-токены, `lucide`-иконки, пилюля под активной вкладкой).

## 4. Модель данных

### 4.1 `users` (доработка)
Уже есть: `telegram_id` (уникальный), `username`, `full_name`, `agency_id`, `role`, `is_owner`,
`is_active`, `session_epoch`, `match_notify`. Языка/телефона в модели нет — это новое.

Добавить поля личного профиля:
- `first_name: str | null` — имя.
- `last_name: str | null` — фамилия.
- `phone: str | null`, **уникальный** — нормализованный номер (E.164). Пока необязателен.
- `phone_verified: bool = false` — подтверждён ли номер (из Telegram-контакта = сразу `true`).
- `language: str = 'ru'` — язык интерфейса (сейчас язык не хранится, идёт через заголовок `X-Lang`).

`full_name` **сохраняем** (его использует существующий код отображения: `created_by_name` и т.п.).
При правке профиля поддерживаем его в синхроне: `full_name = first_name + ' ' + last_name`.
`telegram_id` (уникальный) остаётся способом входа. Поля `user.agency_id` и `user.role`
**остаются на переходный период** для обратной совместимости, но **перестают быть источником
правды** для авторизации (см. 4.2). Их удаление — отдельная будущая задача.

### 4.2 `agency_membership` (существует, миграция 0035 — делаем источником правды)
Одна строка = членство юзера в агентстве:
- `user_id`, `agency_id`, `role` (`agency_admin` | `agent`), `is_owner: bool`, `created_at`.
- Ограничение уникальности `(user_id, agency_id)` — нельзя состоять в одном агентстве дважды.

Роль и владение теперь берутся **отсюда**, а не из `users`.

### 4.3 Задел на будущее (НЕ создаём сейчас)
Отдельная таблица «способов входа/учётных данных» (`auth_identity`: тип telegram/phone/password,
значение, verified) — под сайт/приложение/пароль. Сейчас достаточно `telegram_id` + `phone` в
`users`; таблицу вводим, когда появится SMS/пароль.

## 5. Auth и сессии

- **JWT-claims:** `sub` = внутренний `user_id`, `session_epoch` (мгновенный отзыв — уже есть),
  опционально `act_as_agency_id`.
  - Токен **без** `act_as_agency_id` = личное пространство.
  - Токен **с** `act_as_agency_id` = контекст конкретного агентства; роль берётся из членства.
- **Пара токенов:** короткий рабочий + refresh (`/auth/refresh` уже есть) — основа «оставаться в
  аккаунте» для будущих сайта/приложения.
- **Гварды (дополнить):**
  - `get_current_user` — грузит юзера (личный контекст); больше **не** возвращает 403 при
    отсутствии агентства.
  - `require_agency_member` / `require_agency_admin` / `require_agency_owner` — резолвят активное
    агентство из `act_as_agency_id`, **проверяют членство**, берут роль из членства, применяют
    подписочный гейт (как сейчас). Это обобщение уже существующей проверки владения при `acting`.
- **Суперадмин и `acting`** остаются как есть; механизм лишь распространяется на обычных юзеров.

## 6. Бэкенд: эндпоинты

Новые/изменённые (тонкие роутеры → сервис → репозиторий, слои не перепрыгивать):
- `POST /auth/telegram` — вход всегда успешен → токен личного пространства + бутстрап профиля из
  Telegram (имя, при наличии — номер из контакта). **Убрать** ветку 403 «не в агентстве».
- `GET /auth/me` — личный профиль + список членств (агентства и роль).
- `PATCH /users/me` — обновить `first_name`, `last_name`, `language`.
- `POST /users/me/phone` / `PATCH /users/me/phone` — задать/сменить номер (из Telegram-контакта =
  `phone_verified=true`).
- `GET /me/agencies` — агентства юзера с ролью (для личного хаба).
- `POST /agencies/{id}/enter` — выдать токен-контекст агентства (**обобщить** существующий вход в
  личные агентства на любое членство; проверка членства).
- Возврат в личное пространство — через refresh/новый токен без `act_as_agency_id`.
- `POST /agencies` (создание) — создаёт агентство + членство владельца; **требует подтверждённый
  номер** (иначе понятная ошибка из каталога `errors.MESSAGES`).
- `POST /invites/redeem` — вместо установки `user.agency_id` **создаёт членство** (учитывает
  многоразовые приглашения — уже реализовано).

**Изоляция (священно):** все агентские выборки фильтруются по `agency_id` **из активного
контекста** (`act_as_agency_id`), а не из `user.agency_id`. Данные CRM/объекты остаются
агентство-скоупные.

## 7. Фронтенд

- **Машина фаз** (расширить текущую в `App.tsx`):
  `loading → onboarding → personal → agency` (+ существующие `open`, `reconnect`, `suspended`).
- **Новые экраны:**
  - `LanguagePick` — выбор языка (RU/UZ/EN).
  - `ProfileSetup` — имя, фамилия, «поделиться номером из Telegram» (кнопка `requestContact`),
    номер редактируемый; «продолжить» доступно без номера (номер обязателен лишь при создании
    агентства).
  - `PersonalHub` — карточка профиля, список агентств с ролью, «＋ добавить» (лист: создать /
    вступить по коду); те же кнопки — в «Настройках».
  - `AgencySwitcher` — нижний лист переключения агентств + «Личное пространство».
- **Рабочее пространство агентства** — переиспользуем текущий `Shell` (это и есть нынешнее
  приложение). В шапку добавить: «‹» (выход в личное) и переключатель по названию агентства.
- **Store:** текущий контекст (личное / id агентства), список членств, профиль.
- **Дизайн:** строго существующая дизайн-система (`components/ui.tsx`, токены `index.css`,
  `lucide-react`, пилюля-подсветка нижних вкладок). Никакого инлайн-CSS/сторонних UI-китов.
- **Получение номера в Telegram:** через `requestContact` Telegram WebApp — номер приходит
  подтверждённым.

## 8. Миграция существующих данных (Alembic, не startup-хук)

1. Добавить в `users` поля из 4.1 (`first_name`, `last_name`, `phone` уникальный, `phone_verified`,
   `language`) с безопасными дефолтами/`server_default`.
2. Бэкфилл: `first_name` из существующего `full_name` (фамилия — пусто, если не разделять
   надёжно); `language` = `ru`; `phone` — пусто. `full_name` оставляем как есть.
3. Для каждого юзера с `agency_id` — вставить членство `(user_id, agency_id, role, is_owner)`, если
   его ещё нет. Так у каждого текущего юзера сразу одно агентство в личном хабе.
4. `user.agency_id`/`user.role` не трогаем (обратная совместимость на переходный период).

Ничего не теряется; изоляция не нарушается.

## 9. Тесты (обязательно, pytest + фронт-билд)

- **Изоляция между членствами:** юзер, активный в агентстве A, не видит данные агентства B, даже
  если состоит в обоих (пока не переключится); не-член не может войти в агентство.
- **Роль из членства:** права `agency_admin`/`agent` резолвятся из членства, не из `users`.
- **Онбординг:** новый юзер без членств попадает в личное пространство, а НЕ получает 403.
- **Создание агентства** требует подтверждённого номера.
- **Redeem приглашения** создаёт членство; мультичленство работает (+ уже покрытая многоразовость).
- **Миграция:** существующий одно-агентский юзер получает ровно одно членство с прежней ролью.
- Желательно добавить **HTTP-level** проверки гвардов (аудит отмечал их отсутствие).
- Фронтенд: `npm run build` без ошибок TS.

## 10. На будущее (осознанно отложено)

Вход по SMS-коду (сайт/приложение), логин+пароль, подтверждение номера при смене, точечные сессии
по устройствам, удаление legacy `user.agency_id`, функции мессенджера/соцсети. Модель данных и
токены спроектированы так, чтобы это добавлялось без переделки.

## 11. Риски и внимание

- **Изоляция теперь зависит от `act_as_agency_id` в токене** — критично строго проверять членство в
  гвардах; закрыть это HTTP-тестами.
- Переходный период с двумя источниками (`user.agency_id` vs членство) — держать одно правило:
  **авторизация читает членство**, `user.agency_id` только для совместимости, не для решений.
- Не ослаблять существующую защиту (SSRF, Fernet-шифрование токенов, каталог ошибок, ретраи БД).
