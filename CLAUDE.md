# Architecture

Realty AI — Telegram Mini App SaaS для агентств недвижимости. Мультитенантный: один инстанс обслуживает много агентств, данные строго изолированы по `agency_id`.

**Юзер-центричная модель (пивот 2026-07, в проде).** Всё вращается вокруг человека, а не агентства: любой вошедший получает личный аккаунт (`role='user'`, `agency_id=NULL`) и попадает в **личный хаб** (`frontend/src/screens/Personal.tsx`), откуда сам создаёт агентство (`POST /agencies/register`) или вступает по коду. Один человек может состоять в **нескольких агентствах** с разными ролями — источник правды `agency_memberships` (таблица, миграция `0035`); `users.agency_id/role/is_owner` остаются «домашним» активным членством. Переключение между агентствами — через acting-контекст (`act_as_agency_id`, `POST /agencies/{id}/enter`). **CRM-данные (клиенты/объекты) остаются строго изолированы по `agency_id`** — мультиагентство касается только членства. Суперадмин видит и людей (`GET /superadmin/users` → их агентства/роли + **только объекты**, клиенты приватны). Роль `user` не проходит `require_agency_member` (получает `not_in_agency`), пока не войдёт в агентство.

**Backend** (`backend/`) — FastAPI + SQLAlchemy 2.0 (**синхронный**, `Session`, НЕ async) + PostgreSQL 16, слоистая архитектура:

```
app/
  api/          # роутеры FastAPI, тонкие: разбор запроса → вызов service → schema-ответ
  schemas/      # Pydantic-модели запросов/ответов (граница API)
  services/     # бизнес-логика; здесь и только здесь принимаются доменные решения
  repositories/ # доступ к БД; весь SQL/ORM живёт тут
  db/           # models (SQLAlchemy), session, migrate, retry
  core/         # errors (i18n), monitoring, конфиг-адъютанты
  config.py     # pydantic-settings, единственный источник настроек
```

Поток запроса: `api → service → repository → db`. Слои не перепрыгивать (роутер не ходит в БД напрямую, service не собирает JSON-ответ руками).

**Frontend** (`frontend/`) — React 18 + Vite 5 + TypeScript + Tailwind 3, раздаётся через Caddy. `src/screens/` — экраны, `src/components/` — переиспользуемые компоненты.

**Инфраструктура** — Docker Compose: `db`, `backend`, `backup`, `web` (Caddy :8080), `ngrok` (постоянный HTTPS для Telegram). Прод крутится в WSL2 на офисном ПК `pc1`.

# Folder Structure

- Новый функционал бэкенда = добавь по файлу в `schemas/`, `services/`, `repositories/` (и роутер в `api/`), а не всё в один модуль.
- Модели БД — только в `app/db/models/`, каждая сущность в своём файле, регистрируются через импорт в `app/db/models/__init__.py`.
- Тесты — в `backend/tests/`, один файл на фичу: `test_<фича>.py`.
- Фронтенд: экран = папка/файл в `screens/`; общий UI выносится в `components/`, не дублируется между экранами.

# Coding Standards

**Backend (Python):**
- Форматирование и линт — **ruff**; типы — **mypy**. Прогонять перед коммитом (см. Testing Rules).
- Работа с БД — синхронный `Session` из `app/db/session.py` через `Depends(get_db)`. **Не** тащить сюда async SQLAlchemy — весь код синхронный.
- Весь доступ к данным — через `repositories/`. Сервис не пишет `session.query(...)` сам.
- Настройки — только через `from app.config import settings`. Никаких прямых `os.getenv` в бизнес-логике.
- Ошибки пользователю — через каталог `app/core/errors.py` (`MESSAGES` + `translate`), а не голые строки: интерфейс трёхъязычный (ru/uz/en).

**Frontend (TypeScript):**
- Строгий TS (`tsc` в билде обязателен — билд не должен падать по типам).
- Стили — Tailwind-классы, без инлайн-CSS и сторонних UI-китов.
- Анимации — `framer-motion`, иконки — `lucide-react` (уже в зависимостях, новые не добавлять без причины).

# Naming Conventions

- Python: `snake_case` для функций/переменных, `PascalCase` для классов и моделей, модули — `snake_case`.
- Репозитории — `<сущность>_repo` (напр. `user_repo`), функции в них — глаголы (`get_by_telegram_id`, `create`, `list_for_agency`).
- Pydantic-схемы: `XxxCreate`, `XxxUpdate`, `XxxOut`.
- React-компоненты — `PascalCase.tsx`, хуки — `useXxx`.
- Alembic-миграции — осмысленный slug в сообщении (`alembic revision -m "add_rent_flag_to_objects"`).

# Security Rules

Проект уже прошёл security-ревью (CodeRabbit, аудит) — не откатывай защиту.

- **Изоляция арендаторов — священна.** Любой запрос данных агентства фильтруется по `agency_id`. Никогда не возвращай сущности без проверки принадлежности текущему агентству. Ломается — это критический баг (см. `test_isolation.py`, `test_personal_agencies.py`).
- **Секреты — только в `.env`** (в репо нет, только `.env.example`). Не хардкодить ключи, не логировать их.
- **Google refresh-токены и прочие секреты в БД шифруются Fernet** (`cryptography`). Хранить в открытом виде нельзя.
- **Импорт по ссылке защищён от SSRF** (`test_listing_import_ssrf.py`). При работе с загрузкой URL/фото не убирай проверки адресов и не ходи по внутренним/локальным IP.
- **Телефоны санитизируются** перед сохранением (`test_phone_sanitize.py`).
- Зависимости с CVE-пинами (`python-multipart>=0.0.32`, `pillow>=12.2`) вниз по версии не опускать — комментарии в `requirements.txt` объясняют почему.
- JWT — через `pyjwt`, секрет из настроек; срок жизни и алгоритм не ослаблять.

# Performance Rules

- Никаких N+1: агрегирующие выборки — одним запросом в репозитории (`eager load`/join), а не циклом по строкам.
- Тяжёлые внешние вызовы (Gemini-разбор объявления, рендер страниц через Playwright, импорт фото по httpx) — вне транзакции БД, с таймаутами.
- К БД обращаться через обёртку ретраев (`app/db/retry.py`, `install_db_retry`) — не глотать и не дублировать её логику вручную.
- Долгие/периодические задачи — через планировщик (`app/services/scheduler.py`), а не блокирующие вызовы в обработчике запроса.

# Testing Rules

- Тесты — **pytest**, лежат в `backend/tests/`. Запуск: `cd backend && pytest`.
- Линт/типы перед коммитом: `ruff check backend` и `mypy backend` (кэши `.ruff_cache`/`.mypy_cache` уже в `.gitignore`).
- **Любая новая фича или багфикс сопровождается тестом.** На каждую заметную область уже есть свой файл — расширяй существующий или добавь `test_<фича>.py`.
- Обязательно покрывать: изоляцию арендаторов, права/роли (суперадмин, владелец агентства), импорт данных, платежи, восстановление/удаление.
- Не мокать то, что можно проверить на реальной сессии БД (в `conftest.py` есть фикстуры) — предпочитать сквозную проверку поведения.
- Frontend: минимум — `npm run build` проходит без ошибок TS.

# Git Workflow

- Основная ветка — `main`. Коммиты и сообщения — на русском, в повелительном наклонении («Добавить…», «Исправить…»).
- Один коммит — одно логическое изменение. Не мешать рефакторинг с фичей.
- Не коммитить: `.env`, `/backups/`, `node_modules/`, `.agents/`, `skills-lock.json`, кэши (`.ruff_cache`, `.mypy_cache`, `.pytest_cache`) — всё уже в `.gitignore`.
- **Деплой:** правки → `commit`/`push` в GitHub → на `pc1`: `cd ~/Realty-AI && git pull --ff-only && docker compose up -d --build`. Прод-репозиторий — `~/Realty-AI` (Linux-ФС в WSL2), а НЕ `.bat`-кнопки на Windows. Вход на pc1 (2026-07): Windows-аккаунт **`USER`** (не `adhambek`); мой SSH-ключ сейчас отклоняется → по паролю через PuTTY `plink -hostkey … -pw … USER@100.81.172.20 "wsl -e bash -lc '…'"`. Долгий build рвёт SSH по таймауту Tailscale → запускать деплой оторванно (`setsid … >/tmp/deploy.log 2>&1 &`) и опрашивать лог. Подробности — в памяти проекта.
- Схему БД менять только миграцией Alembic. Одноразовые правки данных — тоже миграцией, НЕ кодом «при старте» (см. комментарий в `app/main.py`).

# Refactoring Rules

- Меняешь поведение — сначала есть/добавь тест, потом рефактори.
- Уважай границы слоёв: логику из роутеров опускай в `services/`, доступ к БД — в `repositories/`.
- Не переписывай рабочие подсистемы (изоляция, планировщик, ретраи БД, каталог ошибок) «на свой вкус» без явной причины — они прошли ревью.
- Мёртвый код удаляй вместе с тестами и упоминаниями в документации.

# Forbidden Patterns

- ❌ Async SQLAlchemy / смешивание sync и async сессий — проект целиком синхронный.
- ❌ Прямой SQL/ORM в роутере или сервисе в обход `repositories/`.
- ❌ Выборка данных агентства без фильтра по `agency_id`.
- ❌ `os.getenv(...)` в бизнес-логике вместо `settings`.
- ❌ Хранение секретов/токенов в БД в открытом виде (только Fernet).
- ❌ Логирование секретов, токенов, `.env`-значений.
- ❌ Ходить по произвольным URL/IP при импорте без SSRF-проверок.
- ❌ Одноразовые миграции данных в startup-хуках вместо Alembic.
- ❌ Понижение версий пакетов с CVE-пинами.
- ❌ Пользовательские сообщения хардкодом мимо каталога ошибок (ломает трёхъязычие ru/uz/en).
- ❌ Инлайн-CSS и сторонние UI-библиотеки на фронте вместо Tailwind.

# Code Review Checklist

Перед тем как считать задачу готовой, проверь:

- [ ] Данные фильтруются по `agency_id`; изоляция арендаторов не нарушена.
- [ ] Слои соблюдены: `api → service → repository → db`, без перепрыгиваний.
- [ ] Настройки берутся из `settings`, секреты не в коде и не в логах.
- [ ] Токены/секреты в БД зашифрованы (Fernet).
- [ ] Внешний ввод (URL, файлы, телефоны) валидируется/санитизируется; SSRF-защита на месте.
- [ ] Изменения схемы оформлены миграцией Alembic.
- [ ] Добавлены/обновлены тесты; `pytest` зелёный.
- [ ] `ruff check backend` и `mypy backend` без ошибок.
- [ ] Frontend: `npm run build` проходит (TS без ошибок).
- [ ] Пользовательские сообщения — через каталог ошибок (обе языковые версии).
- [ ] Нет N+1 и блокирующих внешних вызовов внутри транзакции.
- [ ] Документация обновлена, если менялась архитектура.
- [ ] Ничего лишнего не попало в коммит (`.env`, бэкапы, кэши, `.agents/`).
